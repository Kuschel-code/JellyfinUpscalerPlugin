Contributing

Patches, docs, translations, model catalog entries — all welcome. Here's the short version of how to land a change cleanly.

Repository layout

JellyfinUpscalerPlugin/
├── Api/                  # ASP.NET controllers (the /Upscaler/* surface)
├── Configuration/        # Plugin config UI (HTML/JS in one file)
├── Services/             # VideoProcessor, VideoAnalyzer, HttpUpscalerService, ...
├── ScheduledTasks/       # LibraryUpscaleScanTask, LibraryImageScanTask, cache cleanup
├── Models/               # DTOs for service calls + plugin config POCOs
├── Web/                  # player-integration.js, quick-menu assets
├── docker-service/       # FastAPI service (mirrored into the :latest-* images)
├── site/                 # This website (GitHub Pages)
├── tests/                # xUnit tests
├── docs/                 # MODEL-HOSTING.md, ARCHITECTURE.md, CHANGELOG.md
├── JellyfinUpscalerPlugin.csproj
├── meta.json
├── manifest.json
└── repository-jellyfin.json

Build from source

git clone https://github.com/Kuschel-code/JellyfinUpscalerPlugin.git
cd JellyfinUpscalerPlugin
dotnet restore
dotnet build -c Release

# Run the tests
dotnet test

# Produce a ship-ready publish folder (main DLL + all transitive NuGets)
dotnet publish -c Release -o ./publish
Don't ship the plain bin/Release output. Only dotnet publish emits the full set of transitive NuGet DLLs the plugin needs at runtime. Shipping only the main DLL is what causes the "plugin tombstoned on restart" issue — see Troubleshooting.

Branch & PR flow

  1. Fork the repo, branch from main. Name it fix/ffmpeg-path, feat/arm-images, etc.
  2. One logical change per PR. If you're fixing a bug and adding a feature, that's two PRs.
  3. Include tests for bug fixes. "Write the test that fails without your fix" is the bar.
  4. Update docs/CHANGELOG.md under Unreleased.
  5. Open the PR against main. CI runs dotnet build, dotnet test, and the site linter.

Code style

  • C#: follow .editorconfig. Allman braces, four-space indent, var where the type is obvious.
  • Python (service): ruff + black. Type hints on all public functions.
  • JS: no build step in the plugin — plain ES2020 that runs in Jellyfin's web view. Keep it that way.
  • Log messages use structured logging: _logger.LogInformation("FFmpeg ready: path={Path}", path); — not string interpolation.
  • No fancy abstractions "for the future". CLAUDE.md rule: three similar lines is better than a premature abstraction.

Good first issues

  • Add a translation for a language we don't yet cover — see Web/i18n/.
  • Write a deployment recipe for a platform not listed on the Deployment page.
  • Contribute a ONNX export script for one of the [self-host required] models in the catalog.
  • Add a test to tests/VideoAnalyzerTests.cs covering a failure mode you've hit.

Adding a model to the catalog

If you want a new model available to everyone, not just your local install:

  1. Export to ONNX. Verify it works standalone with onnxruntime-gpu.
  2. Add an entry to docker-service/catalog.json: 
    {
  "id": "my-model-x4",
  "name": "My Model 4x",
  "scale": 4,
  "type": "ESRGAN",
  "vram_mb": 4000,
  "url": "https://example.com/my-model-x4.onnx",
  "sha256": "..."
}
  3. Open a PR. Include one before/after still from a test clip so we can eyeball the output.
  4. If the weights can't be hosted at a stable URL, add [self-host required] to the display name and document the export recipe in docs/MODEL-HOSTING.md.

Reporting bugs

A good bug report includes:

  • Plugin version, Jellyfin version, AI service image tag.
  • Host OS and GPU model.
  • Exact steps to reproduce, including what model/preset is selected.
  • The last 200 lines of the Jellyfin log containing the failure, and the corresponding AI service log.
  • What you expected to happen vs. what actually happened.

"It doesn't work" without the above will get a polite bot comment asking for them, and then usually bit-rot. Front-load the info.

Non-code contributions

  • Docs: every page on this site is in site/. PRs fixing typos, expanding examples, or adding a new recipe are genuinely appreciated.
  • Testing: running the plugin on a platform the maintainer doesn't have (specific NAS, unusual GPU) and reporting what works counts as a contribution.
  • Translations: see i18n notes above.
  • Sponsorship: GitHub Sponsors link is in the README. Funds go to hosting the CDN that backs the model catalog.

Code of conduct

The short version: be kind. We accept contributions from all skill levels — assume good faith, ask clarifying questions, and don't pile-on new contributors for small mistakes. The full Code of Conduct lives in the repo.
